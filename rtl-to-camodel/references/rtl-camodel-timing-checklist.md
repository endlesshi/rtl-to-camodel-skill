# RTL to CAModel Timing Checklist

Use this checklist before implementing or reviewing an RTL-to-CModel migration. It is distilled from real SU CModel vs RTL timing mismatches and should be applied generally.

## Valid/Ready Backpressure

- Treat port non-empty or `valid=1` as only half of a transfer condition.
- Only consume, `Read()`, pop, or `ReleasePacket()` when the modeled transfer has both valid and ready.
- If the destination queue is full or the output port is not ready, hold the input packet/state stable. Do not read and drop it.
- Never implement RTL backpressure as "consume first, decide later".
- When dispatching to a full downstream port, do not pop the source queue before proving the send can occur.

## Queue and Packet Ownership

- Identify exactly which RTL event owns each pop or push.
- A CModel queue pop should correspond to an RTL handshake, accepted issue, committed update, or clock-edge state transition.
- If an operation can fail due to resource pressure, check the resource before mutating source queues.
- Keep data and its dependency bookkeeping consistent: if an instruction is not sent, its wait/set bookkeeping must not be consumed.

## Same-Cycle Register Update Semantics

- RTL nonblocking assignments update in parallel at the clock edge. Decisions in a cycle normally observe old register values.
- Do not let an earlier C++ function call update shared state that a later function call in the same `Clock()` observes as if it were RTL old state.
- For cross-lane, cross-EU, or cross-port shared state, consider two-phase modeling: compute next state from old snapshots, then commit updates.
- Be especially careful with tables, counters, credits, scoreboards, dependency flags, and arbiters.

## Set/Wait and Dependency Tables

- A set generated by one producer in a cycle should not usually satisfy another consumer's wait in that same cycle unless RTL explicitly has combinational bypass.
- If RTL checks wait conditions from the old table and updates the table at the clock edge, CModel must not serially expose producer updates to later consumers in the same `Clock()`.
- Popping a wait flag should not automatically allow the following main instruction to issue in the same cycle unless RTL explicitly permits it.
- Keep wait queue, set queue, done queue, instruction queue, and dependency table updates aligned to RTL event timing.

## Pipeline and Issue Ordering

- Separate current-cycle eligibility checks from next-cycle state changes.
- Avoid code order becoming accidental priority or accidental bypass unless RTL has that priority or bypass.
- For multiple ports/EUs/lanes, verify whether RTL evaluates them in parallel, priority order, round-robin order, or with explicit arbitration state.
- Model bubbles and stalls as first-class timing behavior, not as skipped work.

## Address and Field Computation

- Match RTL alignment and packing rules exactly, including rounding up to pack boundaries.
- Do not replace RTL address calculation with simplified byte-count arithmetic unless the equivalence is proven for all boundary cases.
- Limit opcode-specific field fixes to the same opcode set RTL uses.
- Check instructions whose field layouts differ, especially opcodes that look related but do not carry the same address fields.

## Verification Checklist

For each migrated module, collect evidence for:

- Accepted input cycles: valid, ready, payload, and whether the source was consumed.
- Internal state transitions: queue depths, scoreboard/table counts, register updates, and pipeline stage movement.
- Output cycles: valid, ready, payload, and completion/done timing.
- Stall cycles: what blocked progress and what state was held.
- Boundary cases: full queues, empty queues, simultaneous set/wait, same-cycle done/issue, odd alignment counts, and unsupported opcodes.

Use `xwave` for RTL waveform evidence when proving cycle timing. If waveform or RTL test data is unavailable, report the missing evidence explicitly.
